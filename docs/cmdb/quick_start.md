# 快速入门

通过本快速入门，您可以快速完成 BK-Lite CMDB 的首条落地闭环：定义资产模型、录入或导入实例、建立关系、通过视图与搜索验证结果，再用自动发现持续补齐数据。整个过程不依赖二次开发，适合团队先建立一套“可用、可查、可持续更新”的资产底座。

## 一、开始前准备

在进入操作前，建议先明确以下内容：

1. 需要纳管的首批对象范围，例如主机、网络设备、K8s 资源、数据库或中间件。
2. 每类对象最少需要维护的关键字段，例如实例名称、所属组织、状态、IP、责任人、环境等。
3. 是否已有存量资产表格可用于首次导入，或已有待接入的自动发现目标。
4. 当前账号已具备 CMDB 中模型、实例或采集任务的相应权限。

如果您的目标是尽快推动 CMDB 落地，建议优先选择一类最典型、变化频繁且排障价值高的对象先行纳管，例如主机、网络设备或 K8s 资源。

## 二、建立首个资产模型

第一步不是直接录入资产，而是先定义资产标准。进入 CMDB 后，在模型管理中创建模型分类和具体模型，为后续实例录入、搜索、导入导出和关系展示打好基础。

建议至少先配置以下内容：

*   实例名称与所属组织等基础字段。
*   业务上真正需要检索和筛选的关键属性，例如 IP、状态、环境、责任人、标签。
*   详情页展示更清晰的字段分组。
*   与其他模型的关系类型，例如“属于”“运行于”“连接到”“依赖于”。

如果已有相近模型，可以直接复用模型复制能力，减少重复配置成本；如果多个模型会共享状态、等级、类型等枚举项，建议同步使用公共选项库统一管理。

>
> *   **图表解读/配置逻辑**：模型管理页决定了“资产长什么样、如何展示、如何关联”。这里完成的不是单纯字段录入，而是在建立整个 CMDB 的数据规则。

## 三、录入或导入首批实例

模型准备完成后，即可进入资产数据页面开始录入实例。首次上线通常有两种常见方式：

*   **单条新增**：适合关键对象、试运行场景或模型规则验证阶段。
*   **批量导入**：适合已存在 Excel 台账、需要快速迁移存量数据的场景。

建议先选取一小批数据进行验证，确认字段规则、唯一性约束和组织归属设置都符合预期后，再扩大范围。这样可以避免在模型尚未稳定时进行大批量导入，造成后续回滚或清洗成本。

> **界面指引：**
>
> ![资产列表](https://static.cwoa.net/6c15745c0d35475ea243a7cb3561ec1a.png)
>
> *   **图表解读/配置逻辑**：资产列表页既是实例维护入口，也是批量导入导出和筛选的工作台。模型一旦建好，这里就是日常资产管理的主操作面。

## 四、补齐实例关系并查看拓扑

CMDB 的价值不仅在于保存属性，还在于表达结构。完成实例录入后，建议挑选一组具有上下游依赖关系的对象，建立实例关系并在详情页查看拓扑。

例如：

*   主机与中间件之间建立“安装于”关系。
*   业务系统与数据库之间建立“依赖于”关系。
*   网络设备与接口对象之间建立“连接到”关系。

这一步可以直接验证模型关系定义是否合理，也能让团队更快感受到 CMDB 在排障与影响分析中的价值，而不是把它当成静态台账。

## 五、用资产视图和全文检索验证结果

当模型、实例和关系具备最小规模后，建议立刻用“资产视图”和“全文检索”做一次验收。

建议重点确认以下几项：

*   资产视图中是否能按分类和模型正确看到实例数量。
*   全文检索能否通过名称、IP、关键标识快速命中目标对象。
*   搜索结果中是否能够继续下钻到详情页和关系列表。
*   大小写敏感、本地历史记录等能力是否满足日常检索习惯。

如果这一步已经能支持团队快速找到对象并理解上下文，说明 CMDB 已经从“建模阶段”进入“可用阶段”。

## 六、配置自动发现，让数据持续更新

当手工台账初步可用后，就应该尽快进入自动发现阶段，让 CMDB 从“一次性录入”升级为“持续同步”。

进入自动发现页面后，可按采集对象树选择目标类型，并根据场景配置：

*   采集方式：协议采集或脚本采集。
*   执行范围：接入点、IP 范围或具体实例。
*   安全参数：账户、密码、端口、超时时间。
*   执行策略：周期巡检、审批后写入等。

建议先使用小范围目标验证采集结果，再逐步扩展到更大范围。对于 K8s、网络设备、VMware、数据库等动态对象，自动发现会显著降低人工维护成本，并持续补齐关系与拓扑信息。

> *   **图表解读/配置逻辑**：自动发现按对象树组织采集能力，用户可以从对象类型直接进入任务配置，而不是自行拼装采集脚本和字段映射。

## 七、通过变更记录和采集摘要完成验收

完成一次自动发现或实例批量调整后，建议回到实例详情和任务结果中检查以下信息：

*   采集摘要中新增、更新、删除、关联数量是否符合预期。
*   变更记录中是否能够看见关键实例和关系的变化轨迹。
*   导入、手工维护和自动发现写入后的结果口径是否一致。

如果您已经能够完成“建模 -> 录入/导入 -> 建关系 -> 搜索/视图验证 -> 自动发现 -> 变更留痕”这条链路，就说明 BK-Lite CMDB 已经具备了真正可投入使用的基础能力。

## 八、推荐的首轮落地方式

如果您希望更稳妥地推进上线，推荐按以下节奏实施：

1. 先选 1 到 2 类最关键资产建模，例如主机和网络设备。
2. 先用少量样本实例验证字段与关系设计。
3. 再批量导入存量数据，形成第一版台账。
4. 最后接入自动发现任务，逐步把动态变更纳入系统。

这种方式既能快速看到成果，又能控制模型设计和数据质量风险，比较适合大多数轻量级运维团队的启动路径。

> **注意 / 安全最佳实践：**
> 自动发现建议优先使用最小权限凭据，并在审批写入或正式周期执行前先做小范围验证。这样既能降低敏感信息风险，也能避免错误采集一次性写入大量异常数据。
