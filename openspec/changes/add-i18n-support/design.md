## Context

BlueKing Lite 是基于 Docusaurus 3.8.1 的文档站点，当前全站内容为中文硬编码。`docusaurus.config.js` 已声明 `locales: ['zh-Hans', 'en']`，但无任何英文翻译内容和语言切换入口。

站点包含：
- 50 个 Markdown 文档（19 个模块，autogenerated sidebar）
- 3 个自定义页面（首页、价格、Playground）
- ~15 个 React 组件，含 200+ 处硬编码中文字符串
- 自定义 navbar 组件（MegaMenu、IconNavbarLink、JoinCommunity、Logout）
- 配置层中文标签（navbar items、footer links、tagline）

## Goals / Non-Goals

**Goals:**
- 用户可通过 navbar 下拉菜单切换中/英文
- 默认语言保持中文（`zh-Hans`），英文路径前缀 `/en/`
- 所有 Markdown 文档有对应英文翻译版本
- 所有 React 组件中的用户可见文字支持双语
- navbar、footer、sidebar 标签支持双语
- 构建时同时生成中英文两个 locale 的静态站点

**Non-Goals:**
- 不做三种以上语言支持（仅 zh-Hans + en）
- 不做实时翻译/机器翻译集成
- 不修改后端 API 返回的中文错误消息（Playground API 错误保持原样）
- 不改变现有 URL 结构（中文路径不加 `/zh-Hans/` 前缀）
- 不做翻译管理平台（Crowdin 等）集成

## Decisions

### 1. 使用 Docusaurus 原生 `<Translate>` 组件处理 React 中的硬编码字符串

**选择**: `@docusaurus/Translate` 组件 + `translate()` 函数

**替代方案**:
- 自定义 `useI18n()` hook + JSON 文件 → 不支持 SSG 提取，非标准方案
- react-intl / i18next → 额外依赖，与 Docusaurus 翻译流程不兼容

**理由**: `<Translate>` 是 Docusaurus 内置方案，支持 `docusaurus write-translations` 自动提取翻译 key，SSG 友好，零额外依赖。对于 JSX 中的文本用 `<Translate>` 组件，对于属性值（如 `title`、`alt`）和纯 JS 数据结构中的字符串用 `translate()` 函数。

### 2. 翻译文件结构遵循 Docusaurus 标准约定

```
i18n/en/
├── code.json                                          ← <Translate> 提取的翻译
├── docusaurus-theme-classic/
│   ├── navbar.json                                    ← navbar 标签
│   └── footer.json                                    ← footer 标签
├── docusaurus-plugin-content-docs/
│   ├── current.json                                   ← sidebar category 标签
│   └── current/                                       ← 翻译后的 .md 文件
│       ├── opspilot/
│       │   ├── introduce.md
│       │   ├── feature.md
│       │   ├── quick_start.md
│       │   ├── capabities.md
│       │   └── tutorial.md
│       ├── monitor/
│       ├── cmdb/
│       ├── ... (其余模块同结构)
│       ├── deploy/
│       │   └── docker-compose.md
│       └── faq/
│           └── index.md
└── docusaurus-plugin-content-blog/                    ← 如有 blog 翻译
```

**理由**: 完全遵循 Docusaurus 约定，构建系统自动识别，无需自定义加载逻辑。

### 3. 组件改造策略：数据结构外提 + translate() 函数

对于数据驱动的组件（如 HomepageFeatures、pricing、MegaMenu），采用 `translate()` 函数处理对象/数组中的字符串：

```jsx
// Before
const versions = { basic: { name: '基础版', description: '核心功能，极简部署' } };

// After
import Translate, {translate} from '@docusaurus/Translate';
const versions = {
  basic: {
    name: translate({id: 'hero.version.basic.name', message: '基础版'}),
    description: translate({id: 'hero.version.basic.desc', message: '核心功能，极简部署'}),
  }
};
```

对于 JSX 内联文本用 `<Translate>` 组件：
```jsx
// Before
<div>全栈能力，轻量落地</div>

// After
<div><Translate id="hero.subtitle">全栈能力，轻量落地</Translate></div>
```

### 4. 翻译 ID 命名规范

采用 `组件名.语义路径` 格式：
- `hero.subtitle` — 首页 hero 副标题
- `hero.version.basic.name` — 首页版本选择器基础版名称
- `pricing.plan.community.name` — 价格页社区版名称
- `navbar.docs` — 导航栏产品文档
- `features.classic.title` — 特性展示经典运维标题
- `playground.mlops.anomaly.title` — Playground 异常检测标题

### 5. localeDropdown 位置

放在 navbar `right` 区域，在 GitHub/DeepWiki 图标之前：

```js
navbar.items: [
  // ... 左侧导航项
  { type: 'localeDropdown', position: 'right' },  // ← 新增
  { type: 'custom-icon-link', icon: 'github', ... },
  // ... 其余右侧项
]
```

### 6. 构建和开发脚本

```json
{
  "start:en": "docusaurus start --locale en --host 0.0.0.0 --port 3001",
  "build": "docusaurus build",
  "write-translations": "docusaurus write-translations --locale en"
}
```

`docusaurus build` 默认构建所有 locale，无需额外配置。

## Risks / Trade-offs

- **构建时间翻倍** → 双 locale 意味着两次完整构建。可通过 `docusaurus build --locale zh-Hans` 单独构建加速开发。
- **翻译质量** → AI 辅助翻译可能有术语不一致。通过统一术语表（如 "监控中心" → "Monitoring Center"）缓解。
- **翻译同步维护** → 后续中文文档更新需同步英文版。暂不引入 Crowdin，后期视需要添加。
- **Playground API 错误** → 后端返回的中文错误消息不在前端翻译范围内，英文用户可能看到中文错误提示。→ 接受此限制，后续可在后端添加 i18n。
- **QR 码社区入口** → 英文版的微信二维码对国际用户意义不大。→ 英文版可改为 GitHub Discussions 链接或保留二维码。
